# 模块05 多模态 —— 问题排查与改动记录

> 记录把"图片+文字"多模态演示从一直报错调通的全过程。
> 最终结果：`qwen-vl-plus` 成功返回看图描述。

---

## 一、最初的现象

运行 `mvn spring-boot:run` 时报错：

```
HTTP 400 - {"error":{"message":"Failed to deserialize the JSON body into the target type:
messages[0]: unknown variant `image_url`, expected `text` ...","code":"invalid_request_error"}}
```

含义：服务端收到的消息里带了图片(`image_url`)部分，但**它认为模型只接受文字(`text`)**——也就是请求被一个"纯文本模型"处理了。

起因：OpenAI 无法充值，改用阿里通义千问视觉模型(百炼，OpenAI 兼容接口)。

---

## 二、排查过程(关键是"逐步缩小范围")

### 步骤1：用 curl 绕过 Spring，直接测模型能力

先排除 Spring AI，验证 Key + 地址 + 模型本身能不能看图：

```bash
curl -X POST https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions \
  -H "Authorization: Bearer $DASHSCOPE_API_KEY" -H "Content-Type: application/json" \
  -d '{"model":"qwen-vl-plus","messages":[{"role":"user","content":[
        {"type":"text","text":"这张图里有什么?"},
        {"type":"image_url","image_url":{"url":"<图片URL>"}}]}]}'
```

结果：
- 用**维基百科图**：报 `Download multimodal file timed out`（阿里服务端下载不了那张图）。
- 用**百度图**：✅ 成功返回"一位穿浅色礼服、卷发的女性…"。

> 结论：**Key + 公共地址 + qwen-vl-plus + 可下载的图 = 完全可用**。
> 那么问题一定出在 **Spring AI 实际发出的请求** 与 curl 不一样。

### 步骤2：再测一个"文本模型"收到图片会怎样

```bash
curl ... -d '{"model":"qwen-plus", ...带图片... }'
```
结果：文本模型 `qwen-plus` 直接**忽略图片**回了句 "Hi there"，**并不会**报 `unknown variant`。

> 结论：`unknown variant image_url` 这个特定错误，来自某个**强制只认 text 的端点/模型**(后面证实是 DeepSeek)。

### 步骤3：抓 Spring AI 真正发出去的请求体(决定性一步)

起一个本地 echo HTTP 服务(端口 8799)，把 `base-url` 临时指过去，
拦截 Spring 真实发送的 JSON，看里面的 `model` 到底是什么：

```bash
export SPRING_AI_OPENAI_CHAT_BASE_URL=http://127.0.0.1:8799
mvn spring-boot:run
```

抓到的请求体显示：

```
MODEL = 'deepseek-chat'          ← 真凶！不是我们配的 qwen-vl-plus
part type = 'text'
part type = 'image_url'          ← 图片部分结构其实是对的
```

> 结论：**配置文件里写的 `model: qwen-vl-plus` 根本没生效，实际发的是 `deepseek-chat`。**
> 请求其实被发去了 DeepSeek(纯文本)，所以才报 `unknown variant image_url`。

### 步骤4：查清"为什么模块配置没生效"——Spring Boot 优先级规则

模块 `application.yml` 通过 `spring.config.import` 导入了共享的
`config/spring-ai-common.yml`(其中 chat 指向 DeepSeek)。

Spring Boot 官方规则：

> **Values from the imported file will take precedence over the file that triggered the import.**
> 被 import 进来的文件，优先级**高于**写 import 的那个文件。

所以 common.yml 的 `deepseek` / `deepseek-chat` **反向覆盖**了模块里写的
`dashscope` / `qwen-vl-plus`。原作者注释里"模块覆盖共享配置"的假设方向反了。

### 步骤5：改成自包含后再测，又冒出 404

去掉 import 后，`model` 正确变为 `qwen-vl-plus`，但报 `HTTP 404`。
原因：`base-url` 写成了 `.../compatible-mode/v1`，而 Spring AI 会**自动再拼**
`/v1/chat/completions`，结果变成 `.../compatible-mode/v1/v1/chat/completions`(双 v1)。

修复：`base-url` 只写到 `/compatible-mode`。

### 步骤6：最终验证 ✅

```
【AI 看图回答】这张图片展示了一位女性，她有着精致的面容和优雅的气质……
========== 模块05 演示结束 ==========
```

## 三、最终真正的改动(只动了 2 个文件)

### 1) `application.yml`（核心，3 处）

| 改动 | 改之前 | 改之后 | 原因 |
|------|--------|--------|------|
| 删除 import | 有 `spring.config.import: common.yml` | 整段删除 | 否则 common 反向覆盖模块配置 |
| base-url | `https://api.openai.com` | `https://dashscope.aliyuncs.com/compatible-mode`(不带 /v1) | 换阿里 + 避免双 v1 → 404 |
| model | `gpt-4o-mini` | `qwen-vl-plus` | 换成支持视觉的模型 |
| api-key | `${OPENAI_API_KEY}` | `${DASHSCOPE_API_KEY}` | 用阿里 Key，走环境变量 |

### 2) `MultimodalDemoRunner.java`（1 行）

```java
// 改之前：维基百科图（阿里服务端下载超时）
String imageUrl = "https://upload.wikimedia.org/.../PNG_transparency_demonstration_1.png";
// 改之后：可被服务端下载的图
String imageUrl = "https://gips3.baidu.com/it/u=3886271102,3123389489&fm=3028&...";
```

### 3) 本地环境变量（不在代码里，已写入 ~/.zshrc）

```bash
export DASHSCOPE_API_KEY=sk-ws-...
```

## 四、三个坑总结(给以后的自己/其它模块)

1. **`spring.config.import` 优先级**：被导入的文件优先级**更高**，模块本地配置覆盖不了它。
   需要用不同 provider 的模块，应**不 import 共享配置、自带完整配置**。
2. **base-url 末尾不要带 `/v1`**：Spring AI 会自动拼 `/v1/chat/completions`，否则双 v1 报 404。
   只写到 `https://dashscope.aliyuncs.com/compatible-mode`。
3. **图片 URL 必须能被阿里服务端下载**：否则报 `Download multimodal file timed out`。

> 排查手法记一笔：当"配置看起来没错但行为不对"时，**直接抓真实发出的请求体**
> (本地 echo 服务 / 抓包)，比反复猜测高效得多——本次正是靠它一眼看到
> `model=deepseek-chat` 才定位到根因。
